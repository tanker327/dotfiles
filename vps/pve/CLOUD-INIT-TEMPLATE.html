01Why bother

# the case for cloud-init templates

Without templates

Every VM is artisanal

  • 10+ minutes clicking through the installer per VM
  • Each VM ends up subtly different — forgot agent here, wrong timezone there
  • SSH keys, hostnames, users set manually
  • No way to script "spin up a new box" reliably
  • Fleet drift over time
With cloud-init

Every VM is reproducible

  • qm clone + qm set + qm start → SSH-ready
  • Identical, known-good base for every VM
  • User / IP / SSH keys / hostname / packages declarative
  • Automation-friendly — entire fleet rebuildable from scripts
  • Snapshot the template; pin to a version; roll back trivially

02Mental model

# two Ubuntu "images" that look similar but behave nothing alike

The single most common mistake: grabbing the installer ISO when you wanted the cloud image. They have similar names but are designed for completely different purposes.

Installer ISO

ubuntu-26.04-live-server.iso

~3 GB · graphical / TUI installer · cloud-init not pre-wired

What it is: a bootable installer. You boot it, click through "language → keyboard → partitions → user", and at the end you have Ubuntu installed.

Use for: special-case installs (custom partitioning, encrypted disks), or when no cloud image exists.

Cloud image

ubuntu-26.04-server-cloudimg.img

~600 MB · pre-installed minimal Ubuntu · cloud-init ready

What it is: a frozen post-install Ubuntu disk. Already booted once at Canonical's build farm. You import it and cloud-init does the per-VM configuration on first boot.

Use for: 95% of all VMs. Always reach for this first.

Source URLs:
Ubuntu — cloud-images.ubuntu.com/releases/<version>/release/
Debian — cloud.debian.org/images/cloud/

amd64 vs amd64v3

Ubuntu publishes two server cloud images per release. Use amd64v3 if your host CPU supports it — same Ubuntu, but glibc / OpenSSL / libs are compiled for the x86-64-v3 microarch level (AVX2, BMI1/2, FMA). Expect 5–15% wins in CPU-bound paths.

VariantTargetCompatible CPUs
amd64.imgbaseline x86-64any x86-64 CPU since 2003
amd64v3.imgx86-64-v3Intel Haswell (2013+) / AMD Zen (2017+) and newer

Verify support on the PVE host:

bash · check microarch level
/usr/lib64/ld-linux-x86-64.so.2 --help | grep 'x86-64-v'

If you see x86-64-v3 (supported, searched), you're good. This guide uses amd64v3 throughout — if you're on older hardware, swap the filename for amd64.

04Build the template

# do this once on the Proxmox host

1. Install prerequisites

bash · on PVE host
apt install -y libguestfs-tools

virt-customize lets you modify a disk image without booting it — install packages, run commands, clear identity, all offline. Big speed win over booting and clicking.

2. Download the cloud image

bash
cd /var/lib/vz/template/iso/
wget https://cloud-images.ubuntu.com/releases/26.04/release/ubuntu-26.04-server-cloudimg-amd64v3.img

3. Pre-bake (keep it minimal)

Keep the template lean. The template is the base — everything beyond a working Ubuntu that cloud-init can configure should come from setup-vps.sh after the VM boots.

Pre-bake handles
  • qemu-guest-agent (PVE needs it)
  • Cloud-init datasource pinning (kills 60–90s probe)
  • Machine-id + SSH host key clears
  • Serial console getty
  • curl + ca-certificates so the first command works
setup-vps.sh handles
  • Common tools, mosh, zsh, dev tools
  • Docker + Compose
  • Tailscale
  • UFW + fail2ban (with the rules this VM needs)
  • Per-user: NVM, UV, Bun, Oh My Zsh, dotfiles, Claude Code…
  • Interactive: pick components per VM

Why so minimal? One source of truth (setup-vps.sh). Clean template that boots fast. Component choice per VM — one gets Docker, another doesn't. No drift between the template and the install script.

bash · virt-customize
IMG=/var/lib/vz/template/iso/ubuntu-26.04-server-cloudimg-amd64v3.img

virt-customize -a "$IMG" \
  --install qemu-guest-agent,ca-certificates,curl \
  --run-command 'systemctl enable qemu-guest-agent' \
  --run-command 'systemctl enable serial-getty@ttyS0.service' \
  --truncate /etc/machine-id \
  --delete /var/lib/dbus/machine-id \
  --run-command 'rm -f /etc/ssh/ssh_host_*' \
  --firstboot-command 'dpkg-reconfigure openssh-server' \
  --write '/etc/cloud/cloud.cfg.d/99_pve.cfg:datasource_list: [ NoCloud, ConfigDrive ]'

What each line does:

  • --install qemu-guest-agent,ca-certificates,curl — agent is required for PVE; curl + ca-certificates so curl ... setup-vps.sh | sudo bash works on first SSH.
  • --run-command 'systemctl enable qemu-guest-agent' — must be enabled in the guest, not just in PVE.
  • --run-command 'systemctl enable serial-getty@ttyS0.service' — serial console works on first boot (lifeline when SSH breaks).
  • --truncate /etc/machine-id + --delete /var/lib/dbus/machine-id — every clone generates its own machine-id on first boot (see Tip 6).
  • --run-command 'rm -f /etc/ssh/ssh_host_*' — same idea for SSH host keys.
  • --firstboot-command 'dpkg-reconfigure openssh-server' — regenerates SSH host keys on first boot of each clone.
  • --write /etc/cloud/cloud.cfg.d/99_pve.cfg — restricts cloud-init to NoCloud/ConfigDrive; kills 60–90s of metadata probing.

This adds maybe ~20 MB to the cloud image. Template stays under ~850 MB.

cloud-init = thin base · setup-vps.sh = the actual install

Why clear machine-id and SSH host keys? Cloud images carry identity that must not be cloned. Two VMs with the same /etc/machine-id fight over DHCP leases and corrupt each other's journal. Two VMs with the same SSH host keys cause your known_hosts to scream. Always clear; regenerate on first boot.

4. Create the template VM

bash · qm
VMID=9000

qm create $VMID \
  --name tmpl-ubuntu-26.04-cloudinit \
  --memory 2048 --balloon 0 \
  --cores 2 --cpu host \
  --net0 virtio,bridge=vmbr0 \
  --scsihw virtio-scsi-single \
  --agent enabled=1 \
  --ostype l26 \
  --serial0 socket --vga serial0 \
  --tags template,ubuntu,2604

# import the customized cloud image as the boot disk
qm set $VMID --scsi0 local-lvm:0,import-from=$IMG,discard=on,ssd=1,iothread=1,cache=none

# cloud-init drive — this is where cloud-init reads config from
qm set $VMID --ide2 local-lvm:cloudinit
qm set $VMID --boot order=scsi0

# bake in default SSH key — every clone inherits it
qm set $VMID --sshkey /root/.ssh/authorized_keys

# convert to template and lock
qm template $VMID
qm set $VMID --protection 1

5. Smoke-test with a throwaway clone

60 seconds of pre-flight beats finding a typo at the worst possible moment.

bash · sanity check
qm clone 9000 999 --name test-clone
qm set 999 --protection 0 --ciuser ericwu --ipconfig0 ip=dhcp
qm start 999
sleep 30

# grab the IP from PVE UI or via agent:
qm guest cmd 999 network-get-interfaces | grep ip-address

# try SSH; if it works, cleanup:
qm stop 999 && qm destroy 999 --purge --destroy-unreferenced-disks 1

Why --protection 0 on the clone? Some Proxmox versions copy the template's protection: 1 flag into the clone, which then refuses qm destroy. Clearing it explicitly makes cleanup just work. If you ever hit "cannot remove VM <id> - protection mode enabled", run qm set <vmid> --protection 0 first.

05VM size tiers

# sensible defaults — click one to load it into the builder ↓

"Regular VM" means different things. Here's a tiered set of defaults. All assume the cloud-init template (VMID 9000) is the base — clones override --cores, --memory, and disk size.

Small (2 / 2 / 30) is the right default if you can't decide. Bump if proven necessary, shrink if measured. CPU and RAM are hot-pluggable on virtio; disk grow needs growpart + resize2fs inside the guest.

Universal settings (bake into the template)

Apply to every VM via the template — every clone inherits.

SettingValueWhy
--cpuhostAll host CPU features available; best perf on single-host setups
--balloon0Ballooning steals memory unpredictably under load
--scsihwvirtio-scsi-singleBest perf, iothread support
disk optionscache=none,discard=on,ssd=1,iothread=1DB-safe fsync, TRIM works, perf
--agentenabled=1PVE shows IPs, clean shutdowns, snapshot quiesce
--onboot1Comes back after host reboot
--net0virtio,bridge=vmbr0Paravirt is ~10× faster than e1000
--ostypel26Linux 2.6+; tells QEMU what HW to emulate

Per-VM settings (set on every clone)

SettingWhy per-VM
--cores, --memoryWorkload-dependent (see tier table)
disk sizeWorkload-dependent
--ciuser, --sshkeyPer-user / per-role
--ipconfig0DHCP usually; static for things you need to find reliably
--cicustomRole snippet (base-user, role-db, role-docker, …)
--tagsSearchability — e.g. prod,db or dev,scratch

Capacity planning

Pick a host reserve, then divide what's left across concurrent running VMs.

formula
host_total_ram    = <total RAM, GB>
host_reserve      = 8 GB    # for PVE + ZFS ARC + page cache + headroom
allocatable_ram   = host_total_ram - host_reserve

max_running_VMs  ≈ allocatable_ram / avg_vm_ram

Example on a 96 GB host (reserve 8 GB → 88 GB allocatable):

  • Small (2 GB each) → up to ~44 concurrent VMs
  • Medium (4 GB each) → up to ~22
  • Large (8 GB each) → up to ~11

CPU is far less of a constraint. 2–3× overcommit on vCPUs is comfortable for typical workloads. Only pin / reserve cores for latency-sensitive guests. Disk: thin provisioning means you pay for what's written — watch the thin-pool fill ratio more than per-VM sizes (going over physical disk corrupts every guest).

Static vs DHCP — when to bother

Set a static IP via --ipconfig0 ip=<cidr>,gw=<gw> for VMs you need to find reliably:

  • Database servers
  • Reverse proxies / ingress
  • DNS servers
  • Monitoring / log collectors
  • Anything referenced by hostname or IP in another VM's config

DHCP is fine for everything else — CI runners, scratch VMs, devboxes.

0815 tips for better templates

    
# the things that separate "works" from "delightful"


    

      
TIP 01
Pre-bake packages with virt-customize
Don't make every clone apt install the same packages on first boot. Bake them into the image once.

      
TIP 02
Reserve a VMID range for templates
Use 9000–9099 for templates so they sort together and a typo doesn't qm destroy the wrong VM.

      
TIP 03
Pin CPU type intentionally
--cpu host on single hosts. --cpu x86-64-v3 if you'll cluster. Never the kvm64 default.

      
TIP 04
Enable QEMU guest agent on both sides
PVE side (--agent enabled=1) and guest side (install package, enable service). Without both, snapshots and IP reporting break.

      
TIP 05
Pre-wire your SSH key on the template
qm set 9000 --sshkey ... once. Every clone inherits it; override per-VM when needed.

      
TIP 06 · CRITICAL
Clear machine-id and SSH host keys
Otherwise every clone has the same identity. DHCP leases fight, SSH known_hosts breaks. Regenerate on first boot.

      
TIP 07
Make serial console actually work
Set --serial0 socket --vga serial0 AND systemctl enable serial-getty@ttyS0 in the guest. Your lifeline when SSH is broken.

      
TIP 08
Keep template disk small; resize per clone
Don't bloat the template. Linked clones from a small base are faster. Resize each VM to its actual need.

      
TIP 09
Skip UEFI for Linux cloud images
Ubuntu cloud images target BIOS. Going UEFI adds 4 MB EFI disk per VM and occasional boot quirks. Only use UEFI if you specifically need Secure Boot.

      
TIP 10
Tag your templates
qm set 9000 --tags template,ubuntu,2604. Searchable in UI and CLI. Huge ergonomics win at 30+ VMs.

      
TIP 11
Lock the template against destroy
qm set 9000 --protection 1. Cheap insurance — destroys now require --purge --force.

      
TIP 12
Organize snippets by role
Don't write one giant userdata.yaml. Make base-user.yaml, role-db.yaml, etc. Compose per VM.

      
TIP 13
Sanity-test with a throwaway clone
60 seconds of pre-flight beats finding a typo when you're under pressure to ship.

      
TIP 14
Version templates, don't mutate them
When 26.04.1 ships, make VMID 9001. Keep 9000 around. Easy rollback if 9001 has a regression.

      
TIP 15
Document the template in its description
Source URL, build date, baked packages, usage snippet — all in --description. Future-you will thank you in 6 months.

      - ssh-ed25519 AAAA... ericwu@laptop

10Troubleshooting

    
# common failures, sorted by symptom


    

      Cloud-init doesn't apply settings on first boot
      

        
Check, in order:

        
    
          
  1. qm config <vmid> shows the cloud-init drive (ide2: local-lvm:cloudinit)
    2. 
          
  2. qm cloudinit dump <vmid> user returns sensible YAML
    3. 
          
  3. Inside the VM: sudo cloud-init status --long — if it says disabled, the image was built without cloud-init. Reinstall with apt install cloud-init and re-bake.
    4. 
          
  4. Inside the VM: sudo less /var/log/cloud-init.log for actual errors.
    5. 
        

      

    


    

      SSH connection fails right after clone
      

        
Three common causes:

        
    
          
  • Wrong known_hosts: prior VM had the same IP / hostname. ssh-keygen -R <host-or-ip>.
    • 
          
  • Host keys not regenerated: the template didn't clear them. Confirm with ls -la /etc/ssh/ssh_host_* inside the VM — mtimes should be recent.
    • 
          
  • SSH not yet ready: wait 30s after qm start. Cloud-init runs before sshd is fully up.
    • 
        

      

    


    

      Cloud-init takes 90+ seconds on first boot
      

        
cloud-init is probing AWS / Azure / GCE metadata endpoints that don't exist in your homelab. Restrict to NoCloud:

        

          
bash · inside VM or via virt-customize

          
cat > /etc/cloud/cloud.cfg.d/99_pve.cfg <<EOF
datasource_list: [ NoCloud, ConfigDrive ]
EOF

        

      

    


    

      Disk doesn't expand to full size after resize
      

        
The Ubuntu cloud image includes cloud-initramfs-growroot which auto-grows on first boot. If not:

        

          
bash · inside guest

          
sudo growpart /dev/sda 1
sudo resize2fs /dev/sda1   # or xfs_growfs / for XFS

        

      

    


    

      Cloned VM has same MAC as another (linked clones)
      

        
Full clones get a new MAC; linked clones share the original. Force regenerate:

        

          
bash

          
qm set <vmid> --net0 virtio,bridge=vmbr0,macaddr=auto

        

      

    


    

      I want to start completely over and re-run cloud-init
      

        
DESTRUCTIVE — re-creates the user, re-runs runcmd, etc. Use only on test VMs.

        

          
bash · inside guest

          
sudo cloud-init clean --logs
sudo rm -rf /var/lib/cloud/
sudo reboot
